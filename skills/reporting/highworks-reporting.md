# Highworks Reporting

## Executive-Friendly Length And Tone

Highworks daily reports may be read by a non-developer executive.

- Use concise business Korean. Avoid package names, class names, SQL, mapper names, and raw version strings unless the user explicitly asks to include them.
- Each bullet should usually be 50-90 Korean characters. Add real service, schedule, verification scope, or result context when a line is too short.
- Avoid bullets that run beyond about 100 characters. Merge only closely related work; otherwise split into separate business outcomes.
- `금일` bullets end with completion/progress state and include `(100%)` for completed work.
- `명일` bullets describe planned work without percentage notation.
- Format differs by section: `금일` uses `서비스명 업무영역 ─  내용 (100%)`; `명일` must not use the `서비스명 ─` prefix and should contain only the planned action sentence.

## Purpose

하이웍스 또는 유사한 사내 업무보고 시스템에 붙여넣기 쉬운 일일 보고 양식을 정의합니다.

## 대표 등 비개발자 독자 기준 — 일일 보고 필수 (`금일`·`명일` 공통)

KPA 일일 업무보고 초안은 **대표 및 비개발자 경영층 독자**가 읽는 전제에서 작성한다. **기술·구현 내부만 아는 표현으로 보고 줄을 채우지 않는다.** (휴먼이 *내부 기술 문구 유지*를 명시한 경우만 예외)

**피해야 할 예 (외부 줄에 두지 않음):**

- 코드·설정 식별자: 클래스/메서드/매퍼·SQL·패키지·라이브러리 이름, 버전 표기 문자열 단독 사용
- 구현 디테일: HTTP 헤더/속성 이름, 테스트 전문 용어(예: 스모크 테스트)·CI·스테이징의 내부 코드명 같은 독자가 맥락 없이 이해하기 어려운 용어 나열

**바꿀 때 방향:**

- **무엇이 달라졌나 / 왜 필요한가**를 서비스·사용자·운영 안정 관점 한두 구로 말함
- 환경이 필요하면 **개발 단계 확인 → 검증(사전) 환경 확인 → 최종 사용자 반영** 수준까지만 쓰고, 파이프라인·도구 이름은 생략
- 협업·추적이 필요하면 **업무 과제 카드·협의 채널 담당 부서 결과 공유**처럼 일반화하고, 번호만 던져 의미 없게 만들지 않음

이 절은 `금일`, `명일` **모두**에 적용한다. `금일`에도 Mapper·패치 버전 문자열처럼 내부만 아는 줄은 두지 않는다.

본 문서의 **Reusable Example** 등 과거 입력 예는 교육용일 수 있다. 실제 작성 시에는 **이 절이 우선**한다.

## Filename

```txt
YYYY-MM-DD-report.md
```

## Title Format

```txt
YYYY-MM-DD / B / E / 이름_업무보고
```

예시:

```txt
2026-05-15 / B / E / 김경남_업무보고
```

## Body Format

실제 입력 예(한 줄 패턴):

```txt
## 금일

- 카드톡 운영 ─  카드톡 OX 퀴즈 iOS 플레이 시 ogg 사운드 파일 로드 문제로 해당 파일 mp3로 변경 (100%)
- 달달영어 운영 ─  달달영어 열공 출석부 API 출석 및 응모하기 API 연동 후 데이터 검수 (100%)

## 명일

- 달달영어 API 연동 검수 및 개선
- 카드톡 OX 퀴즈 사운드 리소스 수정 배포
```

- **구분 패턴 (`금일`만):** `서비스/프로젝트 + 업무 성격 ─  세부 내용 (진행도)` 형태를 권장한다. 분리 문자는 입력 예와 같이 ` ─  ` (**앞 한 칸 · 뒤 두 칸** 공백)을 쓴다.
- **`명일` 접두 금지 — 본문만:** `금일`에서 쓰는 **`… ─  …`(앞 카테고리·업무 헤더 + 분리 문자 + 뒷 내용)** 구조를 **명일 줄에는 두지 않는다.** 줄은 `- ` 다음에 **계획 문장 하나만** 작성한다. 서비스·제품·부서 이름이 필요하면 접두가 아니라 **문장 속**에 포함한다 (`달달영어 ─  검수……` 같은 형태 금지).
- **`금일` 진행도:** 각 항목 줄 **맨 끝**에 괄호 진행도를 붙인다. **완료·당일 업무 결과를 보고할 때 기본값은 `(100%)`** 이다. 미완·진행 중이면 예: `(약 70%)`, `(진행중)` 등 실제 상태에 맞게 바꾼다.
- **`명일` 진행도:** **퍼센트나 괄호 진행도를 두지 않는다.**

## 어조: 동작·상태 명사로 종결 (`금일`·`명일` 공통)

- 일반 서술문 종결어미 **`~한다`·`~한다.`·`~함`** 등으로 줄을 끝내지 않는다. 업무 보고서 한 줄은 **실행·결과 상태를 드러내는 명사형 헤드**로 마무리한다.
- 줄 끝에 두기 좋은 예시: **확인**, **검증**, **배포**, **검수**, **점검**, **반영**, **정리**, **인수인계**, **검토**, **협의**, **조치**, **기록**(필요 시 복합: `점검·확인`, `배포·검수`).
- 피할 패턴 예: ~~…확인한다~~, ~~…따른다~~, ~~…문서화함~~, ~~…공유받음~~ → 의미 유지 채 명사형으로 재구성.
- **`(100%)` 위치:** `금일`은 위 명사형 마무리 **바로 다음** `(100%)`를 둔다.

## Rules

- `금일`에는 완료했거나 의미 있게 진행한 업무만 작성합니다.
- `명일`에는 다음 근무일 계획을 작성합니다.
- **`명일` 각 항목(할 일) 본문은 한 줄 기준 최소 50자(공백 포함) 이상**으로 작성합니다. 맥락·검증 범위가 드러나게 쓰며, 짧은 불완전 문장만 두지 않습니다.
- **50자에 미달하는 초안은** 허술한 반복 없이 **대상 기능·일정 창구·점검 항목·선행 조건·검증 범위** 등 필요 정보를 한 줄 안에 추가해 길이를 맞춥니다. 무의미한 수식어만 붙여 글자수를 채우지 않습니다.
- **같은 성격·유형**(예: 같은 시스템, 같은 동사로 시작: `운영 배포`, `검수`, `인수인계` 등) 항목은 **인접하게 묶어** 나열합니다. **`금일`**에서는 가능하면 프로젝트·서비스별로 묶되, **`명일`**에서는 접두 카테고리로 나누지 말고 **본문 순서만** 조정해 같은 주제 인접을 맞춥니다.
- 구현 세부사항보다 업무 목적과 산출물을 우선합니다.
- 이슈가 꼭 필요하면 별도 섹션으로 짧게 작성합니다.
- 제출 전 `Copy/Paste Output`만 복사해도 의미가 완결되어야 합니다.

## Skeleton Template

```txt
YYYY-MM-DD / B / E / 김경남_업무보고

## 금일

- 작업 (100%)

## 명일

- 작업
```

(`금일`의 `작업` 자리에는 실사용 줄 형식으로 대체하고, **`명일`에는 ` ─  ` 접두 없이 본문만** 둡니다. `명일`은 **각 줄 최소 50자** 등은 `Rules` 절 참고.)

## Reusable Example

```txt
2026-05-15 / B / E / 김경남_업무보고

## 금일

- 카드톡 운영 ─  카드톡 OX 퀴즈 검증계 이미지 데이터 마이그레이션 및 검증계 S3 버킷 신규 이미지 리소스 추가 (100%)
- 달달영어 운영 ─  달달영어 열공 출석부 API 출석 및 응모하기 API 연동 후 데이터 검수 (100%)

## 명일

- 달달영어 메인 홈 팝업·배너 노출 규격 개선 차기 반영까지 일정·범위 재확인하고 과제 카드 상태와 사용자 화면 기대 기준 재정리 후 협의 채널 동기화
- 카드톡 OX 퀴즈 사운드 리소스 수정본 검증·운영 배포 일정별 반영 점검 및 배포 결과 확인
```
